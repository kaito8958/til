# TIL（2025-12-11）

### 1. 共有ステージング環境の使い方と注意点を学んだ
- ステージング環境は本番に近い動作確認用の環境で、複数人で共有される。
- 利用時は事前申請し、他の作業と競合しないタイミングでデプロイすることが重要。
- デプロイ後の反映には一定のラグがあることを理解した。

### 2. Git ブランチ運用とステージングデプロイの流れを理解した
- 開発ブランチの変更をステージング用ブランチに取り込み、CI/CD により環境へ自動反映される仕組みを体験した。
- push が環境更新のトリガーとなる流れを理解した。

### 3. フロント側の未保存警告（dirty フラグ）の仕組みを学んだ
- 入力が行われたかどうかをフラグで保持し、未保存状態で離脱すると警告ダイアログを表示するパターンを理解した。
- beforeunload イベントの扱いと、保存操作を契機にフラグをリセットする仕組みを学んだ。

### 4. フロントエンドのビルド工程（npm watch / webpack watch）の重要性を学んだ
- 変更した JS ファイルが反映されなかった原因が、ビルドツールが起動していなかった点にあったと気づいた。
- `npm run watch`（または `make npm_watch`）を起動しておくことで、JS の変更が自動ビルドされる仕組みを理解した。
- フロントの動作確認時には「ビルドプロセスが動いているか」を最初に疑う重要性を実感した。

---

まとめると、今日は **ステージング環境の運用方法、CI/CD の仕組み、未保存警告の実装、JS ビルドフロー** について学びが多い一日だった。
